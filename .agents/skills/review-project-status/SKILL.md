# Project Status レビュー

## 目的

Project の `status.md` 更新後に、次回も同じ品質で読める状態か確認する。この skill はレビュー専用であり、外部情報源の取得や Project の優先度判断は行わない。

## 事前確認

作業前に `projects/README.md` で更新ルールを読み、`create-project-note/templates/status.md` で `status.md` 標準フォーマットを確認する。

レビュー対象は原則として 1 つの Project folder の `status.md` に限定する。対象が不明な場合は、対象 path を短く確認する。

## レビュー観点

- `概要`、`現在の状況`、`ブロッカー`、`未解決事項`、`次に見ること`、`情報源`、`変更履歴` が Project の性質に合う形で揃っているか。
- `概要` の状態、目的、重要度、現時点の判断が、本文と矛盾していないか。
- `現在の状況` が dashboard として読めるか。状態、担当、次アクションまたは判断材料が分かれているか。
- `ブロッカー` で事実と推測が分かれているか。
- 決定済み、完了済み、古くなった懸念が `ブロッカー`、`未解決事項`、`次に見ること` に残っていないか。
- `未解決事項` が、まだ判断・確認が必要な問いだけになっているか。
- `次に見ること` が、次回確認する観測ポイントだけになっているか。
- `情報源` が、次回情報を取得できるリンク付き source として表示名付き Markdown link で残っているか。
- `情報源` に、リンクなしの会話内背景、ユーザーが貼った要約、`YYYY-MM-DD ユーザー提供...`、`会話で確認した内容` のような再取得不能な項目が混ざっていないか。
- `変更履歴` が監査ログではなく、今回変わった判断、状態、ブロッカー、次アクションだけになっているか。
- ログ全文、チケット一覧、メッセージ全文のコピーになっていないか。
- 相対日付ではなく絶対日付で書かれているか。

## 修正方針

明らかな整形、古い項目の移動、見出し名の調整など、レビュー対象ファイル内で安全に直せるものは修正する。

`情報源` にリンクなしの項目がある場合は削除する。会話内背景や貼り付け要約が文脈として必要なら `概要`、`現在の状況`、`ブロッカー`、`未解決事項`、`変更履歴` に要点だけ統合する。リンク先があるはずだが不明な項目は、`情報源` ではなく `未解決事項` に「リンクを確認する」として残す。リンク付き source が 1 つもない場合は、`情報源` を「現時点では未設定。リンク付きの正本が分かったら追加する。」にする。

判断が必要なもの、外部情報源の確認が必要なもの、削除してよいか分からないものは勝手に消さず、レビュー結果の `確認が必要な点` に残す。

## 返答フォーマット

```markdown
## レビュー結果

- 結論:

## 修正したこと

-

## 確認が必要な点

-

## 次に見ること

-
```

修正がない場合は、`修正したこと` に「なし」と書く。

## 境界

- 外部情報源を読みに行かない。
- Project 全体の優先度判断を新しく作らない。
- `status.md` 以外の Project ファイルは、明示的に必要な場合だけ読む。
- Project 本体は local-only なので、レビュー対象の Project ファイルを commit 対象にしない。
