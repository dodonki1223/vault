# 取得情報の分類

## 目的

取得済みの情報を、後続 workflow が扱いやすい形に分類する。

この skill は分類だけを担当する。外部情報を追加取得せず、Project / Note / Linear / GitHub など更新先への反映もしない。

## 入力

分類前に、呼び出し側から渡された入力を確認する。

- 取得済み情報: fetch 系 skill の返答、Slack / Linear / Notion / GitHub などから取得した情報。
- 取得元: Slack / Linear / Notion / GitHub / Docs / その他。
- 取得理由: 何を確認するために取得したか。
- 任意の文脈: Project、Note、Linear issue など、分類に必要な短い背景。

入力が足りない場合でも、追加取得はしない。分類できないものは `分類できなかったもの` に入れる。

## 分類ルール

- 入力に書かれていることだけを使う。
- 事実、決定事項、推測は、それぞれ該当する分類に分けて書く。事実や決定事項の欄に推測を混ぜない。
- 決定事項は、誰かが決めたこと、採用された方針、明示的に合意された内容だけにする。
- 推測は、明示情報から推定したことだけにし、推測である理由を短く添える。
- 未解決事項は、まだ確認・判断・決定が必要な問いにする。
- 次アクションは、取得情報に明示されている action item だけにする。担当、期限、source link があれば添える。推測で次アクションを作らない。
- ユーザー対応待ちは、ユーザー本人の返信、確認、承認、作業が必要なものだけにする。
- 重要リンクは、後続 workflow が辿る価値がある link に絞る。URL だけで置かず、表示名を付ける。
- 同じ内容が複数箇所にある場合は重複排除する。
- ログ全文、コメント全文、議論の全経緯は貼らない。

## 出力フォーマット

```markdown
## 事実

-

## 決定事項

-

## 推測

-

## 重要リンク

-

## 未解決事項

-

## 次アクション

-

## ユーザー対応待ち

-

## 分類できなかったもの

-
```

該当がない分類は空のまま残す。後続 workflow が機械的に扱いやすいように、見出し名は変更しない。

## 各分類の意味

### 事実

取得情報に明示されている状態、発言、更新、期限、担当、結果を書く。

例:

- Linear issue ABC-123 は `In Progress`。
- Slack thread で A さんがレビュー依頼を出している。
- Notion page に対象外として X が記載されている。

### 決定事項

明示的に決まった方針、採用された案、合意済みの内容を書く。

例:

- 仕様 A は案 2 を採用することになった。
- リリース日は 2026-07-02 にする方針で合意された。

### 推測

入力から推定できるが、明示的には確定していないことを書く。

例:

- ABC-123 はレビュー待ちの可能性がある。理由: 担当者からレビュー依頼が出ているが、承認コメントがない。

### 重要リンク

後続 workflow が確認・引用・更新判断に使う link を置く。

例:

- [ABC-123: 対象機能の不具合を修正する](https://linear.app/...)
- [Slack thread: リリース判断の確認](https://...)

### 未解決事項

まだ判断や確認が必要な問いを書く。

例:

- 仕様 A の対象外条件がまだ決まっていない。
- B さんが承認済みか確認が必要。

### 次アクション

取得情報に明示されている action item だけを書く。担当、期限、source link があれば添える。ユーザー本人以外の action item もここに入れる。

例:

- A さんが仕様 A の対象範囲を確認する。期限: 2026-06-28。source: [Slack thread](https://...)
- B チームが PR を作成する。期限: 未定。source: [Google Meet メモ](https://...)

### ユーザー対応待ち

ユーザー本人が対応する必要があるものだけを書く。

例:

- ユーザーが Slack thread の質問に回答する必要がある。
- ユーザーが Linear issue の優先度を承認する必要がある。

### 分類できなかったもの

情報が不足している、文脈が不明、分類先が判断できないものを書く。

例:

- 「対応お願いします」という発言があるが、誰の対応待ちか不明。
- URL だけがあり、何の資料か判断できない。

## 境界

- 外部情報を取得しない。
- ファイルを更新しない。
- Project / Note / Linear / GitHub など、どこに反映するかを決めない。
- `status.md`、README、Linear issue description などへ統合しない。
- 優先度、通知要否、次アクションの最終判断をしない。
- 入力に明示されていない次アクションを新しく作らない。
- 比較対象が必要な `変更点` は扱わない。必要な場合は workflow 側で、既存 note や前回結果と比較して判断する。
