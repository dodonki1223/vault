# Project 必須情報チェックリスト

## 目的

Project 作成時または更新時に、確認できていない重要情報をユーザーに見える形で返す。

不足があっても作業を止めない。ただし、不足している情報を推測で埋めず、作成結果・更新結果・レビュー結果に `不足している情報` または `確認が必要な点` として残す。

## 確認する項目

- 目的と成功条件。
- ユーザーの責任範囲。
- 重要な情報源。
- 対象範囲。
- 除外範囲。
- 完了条件または閉じる条件。

## 具体例

- 目的と成功条件:
  - 何を改善・完了・判断するための Project か。
  - 何が満たされれば「うまくいった」と言えるか。
- ユーザーの責任範囲:
  - ユーザーが判断すること、実装すること、レビューすること、他者に依頼すること。
  - ユーザーではなく別チームや担当者が持つ範囲。
- 重要な情報源:
  - Notion、Docs、社内ドキュメントツールのページ。
  - Linear project、issue、view。
  - GitHub repo、PR、issue、release、workflow run。
  - Slack channel、DM、特定 thread、メンション先。
  - meeting note、録画、議事録、仕様書、設計メモ。
- 対象範囲:
  - 対象の機能、チーム、リポジトリ、issue 群、期間。
  - 追うべき Slack channel や Linear project。
- 除外範囲:
  - 今回は追わない機能、チーム、issue、古い議論。
  - 通知不要な bot 投稿、定型 status、完了済みの情報。
- 完了条件または閉じる条件:
  - release、merge、QA 完了、意思決定完了、handoff 完了など。
  - 定期確認が不要になる条件。

## 不足情報の扱い

- Project 作成時: 分からない項目は `instructions.md` または `status.md` の `未解決事項` に残し、作成結果の `不足している情報` にも出す。
- Project 更新時: 更新後に不足項目を確認し、ユーザーへの返答に `不足している情報` として出す。
- Project レビュー時: 不足項目を `確認が必要な点` に出す。

## 返答例

```markdown
## 不足している情報

- 完了条件または閉じる条件が未確認です。
- 対象範囲と除外範囲の境界が未確認です。

## 次に確認したいこと

- この Project は何を満たしたら完了扱いにしますか？
```
