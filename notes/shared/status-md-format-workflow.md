# status.md 標準フォーマットと更新 Workflow

## 概要

`status.md` は、Project の現在地を 1 分で把握するためのダッシュボードとして扱う。情報源のログを保存する場所ではなく、現在の判断、ブロッカー、未解決事項、次に見ることだけを残す。

- 重要度:

|  |  |  |  |

## Project 種別ごとの表の使い方

### リリース / 不具合対応

| チケット | 状態 | 担当 | リリース判断・次アクション |
|---|---|---|---|

- チケット列には Linear ID とタイトルをセットで書く。
- 状態は Linear の状態だけでなく、リリース判断に必要な補足を短く書く。
- 完了済みのチケットは、現在の判断に必要なものだけ残す。不要になったものは削除するか、変更履歴に移す。

### 横断アサイン / 稼働見通し

| メンバー | 現在持つタスク | 空き見込み | 次に確認すること | 根拠 | 未確認 |
|---|---|---|---|---|---|

- 正本ではなく、アサイン判断材料として使う。
- 「未確認」と「次に確認すること」を混ぜない。
- 古い huddle の暫定アサインは、Linear や本人確認で更新されたら削除する。

### 仕様検討 / 意思決定

| 論点 | 現在の判断 | 担当 | 未解決事項 |
|---|---|---|---|

- 決定済みの仕様は `未解決事項` に残さない。
- 仕様の根拠になる Docs / Notion / PR / Slack thread は `リンク` に表示名つきで残す。

## 更新 Workflow

1. Project フォルダ内の `README.md`、`instructions.md`、`status.md` を読む。
2. 情報源から新しい事実、決定、ブロッカー、担当変更、未対応依頼だけを拾う。
3. `現在の状況` の表を先に更新する。
4. 表に入らない重要な文脈だけを `ブロッカー`、`未解決事項`、`次に見ること` に反映する。
5. 決定済み・完了済み・古くなった項目を `ブロッカー`、`未解決事項`、`次に見ること` から削除する。
6. `変更履歴` に、今回変わった判断、状態、ブロッカー、次アクションだけを短く追加する。
7. 最後に `status.md` を読み返し、「今この Project は何が問題で、次に何を見るべきか」が 1 分で分かるか確認する。

## 更新後チェック

- `概要` の状態と `現在の状況` の表が矛盾していないか。
- `ブロッカー` に、完了済み・解消済み・古い懸念が残っていないか。
- `未解決事項` に、すでに決まった仕様や対応済みタスクが残っていないか。
- `次に見ること` が、次回確認する観測ポイントになっているか。
- 変更履歴が長くなりすぎて、現在状況を読む邪魔になっていないか。
- 日付は絶対日付で書かれているか。
- 外部リンクは表示名つきで、Linear は ID とタイトル、GitHub PR は repo 名または文脈と PR 番号が入っているか。

更新後に情報量が多く、読みきれない状態になっている場合は、[Project 更新後レビュー skill 案](project-update-review-skill-idea.md) の観点でレビューする。特に、削除する情報、圧縮する情報、別 note / 変更履歴へ移動する情報、現在判断として残す情報を分ける。

## 次に見ること

- 既存 project の `status.md` をこのフォーマットに合わせて軽く揃える。
- 特に勤怠リリース系 project で、決定済みの仕様や完了済みチケットが `未解決事項` に残っていないか確認する。
- この Workflow が実運用で使いやすければ、`AGENTS.md` の標準ルールとして維持する。
