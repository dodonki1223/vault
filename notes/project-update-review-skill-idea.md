# Project 更新後レビュー skill 案

## 概要

Project 更新後に `status.md` の情報量が増えすぎて、読みきれない状態になることがある。更新そのものとは別に、更新後レビューを行う skill を発火させ、問題があれば削除・圧縮・移動・構造変更を提案する。

目的は、情報を減らすこと自体ではなく、次に見るべき判断材料にすぐ到達できる状態を保つこと。

## 解決したい問題

- `status.md` がログの蓄積に寄り、現在の状況を読むのに時間がかかる。
- 決定済み・完了済み・古くなった確認事項が残り、未解決事項が信用しづらくなる。
- 重要なブロッカーと背景情報が同じ粒度で並び、優先度が分かりにくい。
- 変更履歴が長くなり、現在状況を読む邪魔になる。
- Slack / Linear / Notion 由来の情報がそのまま残り、Project 固有の判断材料に圧縮されていない。

## skill の責任範囲

候補名: `review-project-update`

この skill は、更新後の `status.md` を読み、以下をレビューする。

- 1 分で Project の現在地が分かるか。
- `概要`、`現在の状況`、`ブロッカー`、`未解決事項`、`次に見ること` が矛盾していないか。
- 決定済み・完了済み・古くなった項目が、現在のブロッカーや未解決事項に残っていないか。
- 情報源ログのコピーが残っていないか。
- 同じ内容が複数セクションに重複していないか。
- 重要度の低い背景情報が、現在判断より目立っていないか。
- 変更履歴に詳細を書きすぎていないか。

この skill は原則としてファイルを直接編集しない。問題がある場合は、修正提案を返す。メインの Codex が提案を確認し、必要なものだけ `status.md` に反映する。

## 返答フォーマット

```markdown
## レビュー結果

- 判定: 問題なし / 要軽微修正 / 要整理

## 問題

- [重要度] 対象セクション: 問題の説明

## 提案

- 削除する:
- 圧縮する:
- 移動する:
- 残す:

## 追加で確認したいこと

- 
```

## 発火タイミング

- `status.md` を更新した直後。
- `status.md` が一定以上長くなったとき。
- `未解決事項` や `変更履歴` が増え続けているとき。
- ユーザーが「読みきれない」「情報量が多い」と感じたとき。

## レビュー観点

### 読みやすさ

- 最初の `概要` だけで、状態・重要度・現時点の判断が分かるか。
- `現在の状況` がダッシュボードになっているか。
- 詳細確認事項が、現在判断より上に来ていないか。

### 鮮度

- 古い日付の確認事項が未解決のまま残っていないか。
- 完了済みチケットが現在ブロッカーに残っていないか。
- 以前の暫定判断が、最新判断と矛盾していないか。

### 粒度

- Slack / Linear / Notion のログがそのまま貼られていないか。
- 1 つの箇条書きに複数の判断が混ざっていないか。
- Project の意思決定に不要な詳細が残っていないか。

### 配置

- 現在の判断は `概要` または `現在の状況` にあるか。
- 調査中の問いは `未解決事項` にあるか。
- 次回確認する観測ポイントは `次に見ること` にあるか。
- 過去の変更は `変更履歴` に短くまとまっているか。

## 他 skill との関係

- `fetch-*` skill は材料を取得する。
- `classify-project-materials` skill は材料を分類する。
- メインの Codex は分類結果を統合して `status.md` を更新する。
- `review-project-update` skill は更新後の `status.md` をレビューし、読みやすさ・鮮度・重複・古い情報をチェックする。

## 次に見ること

- まずは `review-project-update` のチェックリストを手動運用で試す。
- 実際に読みづらい `status.md` を 1 つ対象にして、削除・圧縮・移動の提案が有用か確認する。
- 有用なら skill 化し、`status.md` 更新後に自動で呼ぶ Workflow にする。
